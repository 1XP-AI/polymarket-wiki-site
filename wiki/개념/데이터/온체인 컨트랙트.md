# 온체인 컨트랙트

## 요약

<!-- para: para_001 -->
> Polymarket이 Polygon 체인에서 사용하는 핵심 온체인 컴포넌트와, Copytrade 관점에서 어떤 이벤트를 읽어야 하는지 설명하는 문서.

<!-- para: para_002 -->
Polymarket의 핵심 로직은 **Polygon PoS** (Chain ID: 137) 위의 스마트컨트랙트에서 실행된다. 오프체인 CLOB에서 주문 매칭이 이루어지지만, 실제 토큰 발행·정산·유저 자산 관리는 온체인 컨트랙트가 담당한다.

## 네트워크 아키텍처

<!-- para: para_003 -->
사용자 지갑

```text
└─→ Polymarket 프론트엔드 / API
        ├─→ Gamma / CLOB (오프체인 읽기·주문)
        └─→ Polygon PoS
              ├─ CTF / 결과 토큰
              ├─ Negative Risk 교환 컨트랙트
              └─ Proxy / Safe / 정산 경로
```

<!-- para: para_004 -->
Polymarket이 Polygon을 쓰는 이유는 낮은 가스비, 빠른 블록 타임, EVM 호환성 때문이다. Copytrade 자동화에서도 이 특성이 중요하다. 지갑 이벤트를 추적할 때 지연이 짧고, 트랜잭션 단위 검증을 비교적 낮은 비용으로 반복할 수 있기 때문이다.

<!-- para: para_005 -->
CTF(Conditional Token Framework)는 결과 토큰의 split, merge, redeem 흐름을 설명하는 핵심 모델이다. 하나의 담보를 결과 토큰 묶음으로 나누고, 다시 합쳐 원래 담보로 되돌릴 수 있다는 점이 예측 시장 구조의 기반이 된다.

<!-- para: para_006 -->
Negative Risk 계열 마켓은 일반적인 Yes/No 마켓과 주문 경로가 다르므로 별도 컨트랙트와 라우팅 규칙을 가진다. 따라서 Copytrade 시스템은 단순히 "체결가와 수량"만 따라가면 안 되고, 어떤 마켓이 일반 구조인지 Negative Risk 구조인지 먼저 구분해야 한다.

<!-- para: para_007 -->
Copytrade에서 온체인 컨트랙트를 읽는 목적은 두 가지다. 첫째, 실제 토큰 이동과 포지션 상태를 검증해 API 지연이나 집계 오류를 보완하는 것. 둘째, 특정 지갑이 어떤 결과 토큰을 얼마나 보유했는지 복원해 리더 트레이더의 행동을 더 정확히 해석하는 것이다.

## 열린 질문

- Polygon PoS에서 Polymarket의 CTF(Conditional Token Framework) 관련 온체인 컨트랙트들이 split, merge, redeem를 각각 어떤 이벤트와 상태 변화로 기록하는지, 실제 트랜잭션 예시를 기준으로 추적해보라?
- Negative Risk 계열 마켓과 일반 Yes/No 마켓이 온체인 라우팅과 컨트랙트 호출 경로에서 어떻게 다른지, 차이를 판별할 수 있는 이벤트 시그니처와 주소 패턴을 조사하라?
- Copytrade 시스템이 프론트엔드/API 데이터 대신 온체인 컨트랙트를 읽어 포지션과 자산 보유량을 복원할 때, 어떤 이벤트들을 우선 수집해야 정확도가 높아지는지 정리하라?
