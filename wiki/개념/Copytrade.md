# Copytrade

> 리더보드 트레이더를 따라 거래하는 전략 — 개념, 계량적 평가, 백테스트, 실행, 모니터링을 아우르는 통합 문서

## 요약

Copytrade는 리더보드 상의 검증된 트레이더를 추종해 그들의 포지션 변화를 자동으로 복제하는 전략이다. 이 문서는 카피트레이드의 개념적 기반과 계량적 평가 지표, 백테스트 설계, 실행·슬리피지 고려, 운영 모니터링 체크리스트 및 구현 템플릿을 제시한다.

## 핵심 데이터 소스

### 1. 리더보드 (Leaderboard)

```
GET https://data-api.polymarket.com/v1/leaderboard
파라미터: window=all|daily|weekly|monthly, limit, offset
```

- 수익률 기준 트레이더 순위
- 트레이더 지갑 주소 획득 가능
- 기간별 필터링 (전체/일간/주간/월간)

### 2. 트레이더 프로필

```
GET https://gamma-api.polymarket.com/public-profile?address={wallet}
```

- 프로필 정보, 거래 통계

### 3. 트레이더 포지션

```
GET https://data-api.polymarket.com/positions?address={wallet}
```

- 현재 보유 포지션 전체 조회
- 마켓별 토큰 수량, 방향 (Yes/No)

### 4. 트레이더 거래 내역

```
GET https://data-api.polymarket.com/trades?address={wallet}
```

- 과거 거래 기록
- 체결 가격, 수량, 시간, 방향

### 5. 실시간 감지 (WebSocket)

- Market Channel로 특정 마켓의 거래 실시간 수신
- 특정 트레이더의 거래를 필터링하려면 trades 폴링 또는 온체인 이벤트 구독

## Copytrade 파이프라인

```
1. 리더보드에서 상위 트레이더 선별
   ↓
2. 선별된 트레이더의 포지션/활동 모니터링
   ↓
3. 새로운 거래 감지
   ↓
4. 동일 마켓에 동일 방향으로 주문 생성
   ↓
5. 포지션 관리 (익절/손절/청산 따라하기)
```

## 핵심 개념

- 목표: 숙련 트레이더의 의사결정에서 발생하는 알파(정보 우위)를 재현하되, 복제에 따른 비용·리스크를 계량적으로 통제한다.
- 접근법: 트레이더 행동을 이벤트(포지션 진입/청산)로 감지 → 필터링(유의미성·중복노출·유동성) → 사이징 규칙 적용 → 실행 → 모니터링으로 루프 완성.
- 성공적인 카피트레이드는 트레이더 선정(승률, 위험 프로필), 포지션 스케일링, 진입/청산 규칙, [[전략/Copytrade/리스크-관리|손절(스톱로스) 전략]]을 포함한다.
- 리더보드의 빈도(거래 빈도)와 포지션 사이즈를 분석하여 자동 복제 비율을 결정해야 하며, CLOB 기반 주문집행의 슬리피지와 수수료를 시뮬레이션하는 것이 필수적이다.

## 계량적 선택·평가 지표

1. 일관성 지표
   - 최근 N일(예: 30/60/90) 누적수익률과 표준편차
   - 연속 베팅 시 승률(Win Streak / Loss Streak) 분포
2. 샘플 크기 및 신뢰성
   - 트레이더가 같은 타입(마켓)에서 수행한 거래 수
   - 거래당 평균 노출기간
3. 선호 마켓 유형
   - 정치/스포츠/크립토 등 마켓별 선호도 — 시장 특성에 따른 성과 차이 고려
4. 알파 신호 강도
   - 트레이더의 평균 정보계수(IC): 트레이더 신호와 이후 수익 간의 상관계수
   - 샤프/Sortino 비율 (수익의 리스크조정 성과)
5. 유동성·임박성·정책 리스크
   - 시장별 평균 호가 스프레드·잔존유동성(상위 N 오더) 측정
   - 이벤트 마감까지 남은 시간(임박성)으로 가중치 조정
6. 중복노출(클러스터링)
   - 동일 주제/이벤트에 여러 팔로우로 인한 포지션 집중도

## 백테스트 설계 (권장)

- 데이터 필요
  - 리더보드 스냅샷(시간별), 마켓 상태(당시 가격·유동성), 체결/청산 로그
  - raw/fetched 에 저장된 스냅샷 파일을 사용
- 시뮬레이션 구성
  1) 이벤트 재현: 트레이더 A의 포지션 변경을 시점 t로 고정
  2) 실행 모델: 실제 마켓 유동성·수수료·슬리피지 모델 반영
  3) 사이징: 비율 기반(계정자본 대비) 또는 목표 델타 기반
  4) 평가: 거래비용을 뺀 순수 알파, 누적수익 곡선, 최대 낙폭(MDD), 회복기간
- 민감도 분석
  - 슬리피지, 수수료, 추종 지연(latency)을 ±범위로 변동시켜 민감도 표 작성

## 실행(운영) 고려사항

- 레이턴시 최적화
  - 리더보드 이벤트 탐지 → 신속히 주문 제출까지의 경로 최적화
  - [[기술/API/Polymarket-API-레퍼런스]]와 폴링의 하이브리드 사용
- 주문 유형
  - 시장가 주문은 빠르지만 슬리피지 위험 증가
  - 제한가+IOC/FOC 혼합으로 제출 후 미체결 시 대체 전략
- 실패 모드
  - 주문 거부, 부분 체결, 마켓 상태 급변에 따른 자동 재평가

## 추종 규칙 예시(의사코드)

- Detect: 트레이더 A가 market X에 "Yes" 노출을 신규로 구축했을 때
  1) if liquidity(X) < threshold -> skip
  2) compute exposure = min(max_size, account_capital * target_pct)
  3) check duplicate_exposure(cluster_key) -> if > limit then reduce/skip
  4) submit order with price_control (limit or pegged)
  5) on partial_fill -> attempt completion up to slippage_limit

## 운영 사례 및 사이징 예시

- 사례 A: 소형 마켓(유동성 낮음)에서의 안전한 접근
  - 전제: 계정자본 10,000 USD, 목표 카피비율 target_pct = 0.5% → target_trade_size = 50 USD
  - 시장 유동성 검사: 상위 오더의 잔존 유동성이 200 USD일 때, 단일 진입으로 시장 영향이 크므로 분할 진입(예: 5회, 10 USD씩) 권장
  - 비용 예시: 예상 슬리피지 0.5% + 수수료 0.2% → 총 거래비용 약 0.7% 적용

- 사례 B: 임박성 높고 유동성 양호한 대형 마켓
  - 같은 계정자본 기준 target_pct를 2%로 상향 적용 가능, 단 거래 전 리스크 알림과 상한(예: max_size)을 확인

- 사이징 계산(간단)
  - account_capital * target_pct = raw_size
  - adjusted_size = min(raw_size, max_size, liquidity_limit)
  - 분할 횟수 = ceil(adjusted_size / intended_slice_size)

- 운영 팁
  - 슬리피지·부분체결 이력에 따라 트레이더별 가중치 보정
  - 동일 이벤트에 대한 다중 팔로우는 클러스터링 키로 집계하여 포지션 상한 적용

## 모니터링 & 알림

- 성능 알림
  - 1일/7일 급격한 성과 저하 → 자동 팔로우 중지 플래그
- 리스크 알림
  - 누적 손실(계정 %) 초과 시 휴지 상태로 전환
- 무결성 검사
  - 리더보드 데이터와 실행 로그의 타임스탬프 불일치 감지

## 구현 체크리스트

## 추가 심화: 데이터 예시, 백테스트 파라미터 샘플, 구현 템플릿

### 데이터 예시
- 리더보드 스냅샷(예):
  - {timestamp: "2026-03-01T12:00:00Z", trader_id: "0xabc...", market_id: "mkt123", side: "Yes", size: 75.0, price: 0.62}
- 마켓 유동성 스냅샷(예):
  - {timestamp: "2026-03-01T12:00:00Z", market_id: "mkt123", best_bid: 0.615, best_ask: 0.625, top_n_liquidity: 500.0}

이런 스냅샷들을 시간순으로 정렬해 이벤트 기반 시뮬레이션을 수행한다.

### 백테스트 파라미터 샘플
- latency_ms: [0, 100, 500]
- slippage_pct: [0.0, 0.2, 0.5, 1.0]
- fee_pct: [0.1, 0.2, 0.3]
- slice_size_usd: [10, 25, 50]

각 조합에 대해 누적수익·MDD·회복기간을 기록해 민감도 표를 만든다.

### 구현 템플릿 (Python 의사코드)

```python
# event-driven follow logic (pseudo)
for event in leaderboard_events:
    if not passes_filters(event):
        continue
    target = compute_size(account_capital, event, params)
    slices = split_into_slices(target, params['slice_size_usd'])
    for s in slices:
        res = submit_order(market_id=event.market_id, side=event.side, size=s)
        handle_partial_fill(res)
    record_execution(event, res)
```

> 실제 구현에서는 주문 재시도, 가격 보호(price guard), 재진입 제한(reentry cooldown) 등을 반드시 포함해야 한다.

### 운영 지표 및 권장 임계값 (예시)
- 즉시 중지(Stop-follow) 조건:
  - 연속 손실 5회 또는 7일 누적 손실 > 5% 계정
  - 트레이더의 최근 30일 거래수 < 10 (신뢰도 부족)
- 경고(Warning) 조건:
  - 부분체결 비율 > 30% (시장 충격 또는 유동성 부족)
  - 평균 슬리피지 > 0.5% (실행 정책 재검토 필요)

## Q&A: 자주 묻는 질문

### Q1. Copytrade를 바로 시작해도 되나요?

직접 시작하기 전에 리스크 관리 규칙과 선별 체크리스트를 먼저 마련하세요. 특히 포지션 크기 제한, 스톱로스, 다수 트레이더 분산 복제 전략은 초기에 필수입니다. 단일 트레이더에 과도하게 의존하지 마세요.

### Q2. 어떤 트레이더를 따라야 하나요?

정량적(수익률, 승률, 거래 빈도, MDD) + 정성적(마켓 선호, 활동 기간, 포지션 크기) 기준을 조합해 선별합니다. 리더보드 상위권은 후보가 되지만, 최근 활동과 일관성도 반드시 확인하세요. 자세한 선별 체크리스트는 아래 '초보자용 입문 체크리스트' 섹션을 참고하세요. 초기 후보 발굴과 프로필 분석은 [[전략/Copytrade/트레이더-프로필-분석]]에서 다룹니다.

### Q3. 실시간 복제는 어떻게 처리하나요(지연 문제)?

감지 → 주문 생성 사이의 지연(latency)을 최소화하려면 가능한 한 API 폴링 대신 WebSocket 이벤트와 즉시 제출 가능한 주문 라우팅(사전검증된 가격 한도 주문)을 사용하세요. 플랫폼 구조와 실시간 데이터 처리 방법은 [[개념/Polymarket-개요]]와 [[기술/WebSocket-실시간데이터]] 문서를 함께 확인해 구현 세부사항을 검토하세요. 슬리피지와 유동성 문제를 고려해 진입 크기를 작게 설정하는 것이 유리합니다.

### Q4. 수수료와 슬리피지 때문에 수익이 감소하는데요?

예: 카피 시 대부분 테이커 수수료를 내게 되므로, 수수료와 예상 슬리피지를 반영한 비용 모델을 만들어 기대수익을 조정해야 합니다. 작은 포지션으로 여러 트레이더를 분산하면 비용 대비 이득을 안정화할 수 있습니다. 관련 내용은 [[전략/Copytrade/실행-비용]] 및 [[개념/슬리피지]] 문서를 참조하세요.

### Q5. 트레이더가 갑자기 전략을 바꾸면?

자동 모니터링(성과 점수 하락, 포지션 패턴 변화 감지)을 통해 일정 임계치(예: 누적 수익률 -10% 또는 승률 20% 감소) 이하로 내려가면 자동으로 팔로우를 중단하도록 설정하세요. 또한 오프라인(수작업)으로 정기 검토를 권장합니다.

### Q6. 초보자가 실전에서 실수하기 쉬운 부분은?

1) 포지션 과다집중 2) 수수료·슬리피지 미반영 3) 표본이 적은 트레이더 추종 4) 백테스트 없이 실거래 적용 — 이 네 가지를 경계하세요.

## 초보자용 입문 체크리스트

> 리더보드에서 누구를 따라 거래할지 결정하고 기본 리스크를 관리하는 빠른 시작 가이드

### 핵심 원칙
1. **팔로우 가치 평가** — 트레이더의 일관성(기간, 승률), 포지션 크기, 선호 마켓(정치/스포츠/크립토) 확인. [[전략/Copytrade/팔로우-선별]] 참조.
2. **리스크 한도 설정** — 총 자금의 1~5%를 개별 트레이더 노출 상한으로 설정. [[전략/Copytrade/리스크-관리]]의 헤징 기법 병행 권장.
3. **진입/청산 규칙 명문화** — 트레이더 진입 직후 무조건 따라가지 말고, 스프레드·유동성·청산 패턴을 고려해 타이밍 결정.

### 운영 사례
- **사례 A (소액, 초보자):** 자금 $1,000, 개별 트레이더 노출 2%($20), 7일 승률 60%+ 시 검토 후 진입, 손절/유동성 급감 시 즉시 청산.
- **사례 B (중간, 자동화 일부):** 자금 $10,000, 개별 노출 1% 기본·고확신 3% 상향, 주문 3회 분할 실행, 주간 리더보드/PnL 평가.

### 사이징 계산
1. **고정비율(Fixed Fractional):** 계정 × X% (예: $5,000 × 2% = $100/트레이더)
2. **변동 리스크 기반:** 트레이더 과거 변동성에 가중치 (30일 MDD 10% → 노출 0.5~1.0%로 축소)

### 샘플 실행 워크플로
1. 리더보드 스냅샷 수집(주간)
2. 상위 50명 필터링 → 지표 표준화(z-score)
3. 상위 10명으로 압축 후 프로필·포지션 API 조회
4. 샌드박스에서 100회 백테스트(슬리피지 0.5%, 지연 1~5분 가정)
5. 소규모 롤아웃(각 트레이더에 포트폴리오의 0.5~1.0% 할당)
6. 실시간 모니터링(포지션 불일치, 누적손실 임계치) → 자동 중단

### 백테스트 결과 해석 가이드
- **기대값:** CAGR 12%, MDD 8% 이면 합리적
- **실패신호:** 백테스트에서 리밸런싱/지연을 반영했을 때 성능이 급락하면(예: CAGR -5% → 제외)
- **노이즈 확인:** 소규모 트레이더(거래건수 < 50)에서 나온 우연적 성과는 신뢰 낮음

### 빠른 시작 체크리스트
- [ ] 리더보드 상위 50 스냅샷 확보
- [ ] 상위 10명에 대해 포지션·프로필 추가조회
- [ ] 백테스트(거래비용·지연 포함) 수행
- [ ] 포지션당 노출 제한 설정(예: 1~2% 목표)
- [ ] 자동 중단 규칙 구성(성과·포지션 불일치)

### 초기 선별 체크리스트 (A~E)

각 항목을 통과한 트레이더만 다음 단계로 넘긴다:

- **(A) 리더보드 순위**: 최근 7일 평균 순위가 상위 20% 이내
- **(B) 최근 30일 누적 수익률**: 양수이며 거래횟수 >= 10
- **(C) 평균 포지션 유지기간**: 사용자의 허용범위(예: 1~7일)와 일치
- **(D) 포지션 사이즈 변동계수(CV)**: < 0.8 (일관성 척도)
- **(E) 소셜/프로필 공개성**: 전략/포스트 링크 존재 (트위터·블로그)

선별 예시: 50명 → A~D 적용 후 ~18명 → E 적용 후 ~7명 → 샘플 추종 시뮬레이션으로 최종 3명 선정.

### 샘플 점수화 규칙 (가중치 예시)

- 리더보드 랭크(0~30): 상위 5%=30, 5~20%=20, 20~50%=10
- 최근 30일 수익률(0~30): 상위 분위수 기반 가중치
- 승률 및 샘플 크기(0~20)
- 정성(공개성·투명도)(0~20)

총점 70점 이상을 우선 후보로 고려. 가중치는 운영 목적(단기/중기)에 따라 조정.

### 실행 워크플로우
1. 리더보드에서 후보 선별 → 2. 정량 지표(승률, 드로다운) 점수화 → 3. 사이징·주문 규칙 결정 → 4. 자동/수동 진입 → 5. 주간 성과 점검 → 6. 헤징·분산·추종 중단

## 기술적 제약

1. **Polymarket에는 공식 Copytrade 기능이 없음** — 직접 구현 필요
2. **지오블로킹**: 미국 등 일부 지역에서 접근 제한
3. **Rate Limit**: API 호출 빈도 제한 있음
4. **온체인 지연**: 블록 확인 시간 존재
5. **Gasless 한계**: Relayer 방식이므로 트랜잭션 순서 보장 어려움

## 리스크 관리

### 주요 리스크
1. **시간 지연 (Latency)**: 감지 → 주문 사이 시간차로 불리한 가격에 진입
2. **유동성**: 카피 대상이 대규모 주문 → 슬리피지 발생
3. **전략 변경**: 트레이더가 갑자기 전략 변경
4. **수수료 침식**: 카피 트레이더는 대부분 Taker → 수수료 부담
5. **역선택**: 과거 성과 ≠ 미래 성과

### 대응 전략
- **다수 트레이더 분산 카피**: 1명에 올인하지 않음
- **포지션 크기 제한**: 계정자본 대비 비율에 상한 적용 (max_size)
- **스톱로스**: 카피 대상이 청산하지 않더라도 자체 손절
- **성과 모니터링**: 일정 기간 성과 미달 시 자동 중단
- **딜레이 분석**: 진입 시점의 가격 vs 감지 시점 가격 비교 (참조: [[전략/Copytrade/딜레이-분석]])
- 손절 규칙: 연속 손실 횟수 또는 기간 기준 손실률 초과 시 팔로우 정지
- 포트폴리오 다양화: 단일 마켓·동일 클러스터에 대한 집중도 제한
- Negative-Risk 전략: 자본 효율적 헤징을 위해 음의 상관관계 포지션 활용 고려

## 관련 섹션

### 온체인 컨트랙트

> 폴리마켓의 온체인 토큰·정산 메커니즘과 서명/체결 흐름.

Polymarket은 Polygon 체인 위에서 결과 정산과 토큰 수령을 처리한다. CTF 토큰의 Split/Merge, 승리 토큰 Redeem 과정은 온체인 트랜잭션으로 실행된다. 서명된 주문은 CLOB를 통해 제출되고 체결 시 온체인 정산이 이루어진다.

- [[개념/CTF]] — CTF 토큰의 Split/Merge/Redeem 흐름
- 주문 ABI 및 호출 예제: Polymarket ABI는 Polygon Explorer에서 확인
- 온체인 데이터를 통한 포지션 검증

### 마켓메이커 리베이트

> Maker 리베이트 프로그램과 오더북 기반 수수료/리베이트 메커니즘 설명

Polymarket의 Maker는 오히려 리베이트를 받는 구조를 운영한다. 리베이트는 유동성을 제공하는 참여자에게 지급되어 오더북의 스프레드를 좁히고 거래 깊이를 증가시킨다. 마켓메이커가 고려할 항목: 스프레드 설정, 주문 깊이, 주문만료 정책, 체인상 가스 정책 및 리베이트 수령 조건.

리베이트 프로그램의 상세 조건(지급 기준, 기간, 최소 거래량 등)은 플랫폼 문서와 보상 프로그램 공지를 참조해 업데이트해야 한다.

- 리베이트는 마켓 메이킹을 장려하여 스프레드를 줄이고 거래 비용을 낮춘다.
- Copytrade 관점에서 높은 Maker 활동을 보이는 트레이더의 전략은 유동성 제공 성향을 반영할 수 있다.

### CTF 개념

CTF 관련 용어와 Copytrade에서의 의의, 리스크를 정리한다. 키워드: CTF 조건부 거래 담보 프레임워크 리스크 헤지 청산 규칙
