# 온체인 컨트랙트

> 한 줄 요약: Polymarket이 Polygon 체인에서 사용하는 온체인 컴포넌트(CTF 토큰 발행·스플릿/머지·정산, NegRiskExchange, Proxy/Safe)의 구조와 Copytrade에서의 온체인 데이터 활용법을 설명합니다.

## 내용

Polymarket의 핵심 로직은 **Polygon PoS** (Chain ID: 137) 위의 스마트컨트랙트에서 실행된다. 오프체인 CLOB에서 주문 매칭이 이루어지지만, 실제 토큰 발행·정산·유저 자산 관리는 온체인 컨트랙트가 담당한다.

## 네트워크 아키텍처

```
사용자 브라우저 ─→ Polymarket 프론트엔드
                      │
                      ├─→ Gamma API (Gamma): 이벤트/마켓/오더북 읽기
                      ├─→ CLOB API (clob.polymarket.com): 주문 작성/취소
                      │
                      └─→ Polygon PoS (Chain 137): 온체인 정산
                              │
                              ├─ CTF 토큰 컨트랙트
                              ├─ NegRiskExchange 컨트랙트
                              └─ Proxy/Safe 컨트랙트
```

### 왜 Polygon?

- 낮은 가스비: 수백~수천 건의 소규모 거래에 적합 (평균 $0.001 미만)
- EVM 호환성: 기존의 이더리움 도구(MetaMask, Hardhat, ethers.js)를 그대로 사용
- 빠른 블록 타임: ~2초 블록으로 실시간 거래 경험 제공
- Polymarket의 모든 정산과 토큰 발행은 이 체인 위에서 최종적으로 확정된다

## 핵심 컨트랙트

### 1. CTF (Conditional Token Framework)

Gnosis(구 Augur v2)에서 개발한 ERC-1155 기반 토큰 표준. 예측市場的 결과를 토큰화하여 거래·정산하는 기반이다.

**세 가지 핵심 연산:**

| 연산 | 입력 | 출력 | 설명 |
|------|------|------|------|
| **Split** | USDC.e collateral | Yes + No 토큰 | 담보를 결과 토큰 쌍으로 분할 |
| **Merge** | Yes + No 토큰 쌍 | USDC.e collateral | 결과 토큰 쌍을 담보로 재결합 |
| **Redeem** | 승리 토큰 | USDC.e | 정산 후 승리 토큰을 담보로 인출 |

**Split 예시 (유사 코드):**

```solidity
// 의사코드: ConditionalTokens.splitPosition
function splitPosition(
    IERC20 collateral,      // USDC.e
    bytes32[] conditionIds, // 조건 ID 배열
    uint256[] partition,    // [1, 2] (Yes/No 인덱스)
    uint256 amount          // 분할할 collateral 수량
) external;
```

- 1 USDC.e를 deposit → Split → 1 Yes 토큰 + 1 No 토큰 생성
- Merge: 1 Yes + 1 No → 1 USDC.e 복원 (손실 없음, 가스비만 차감)
- Redeam: 승리 토큰 보유 → 정산 후 full USDC.e 인출
- 패배 토큰: 가치 0 → Redeem 시 아무것도 받지 못함

CTF는 토큰의 **조건부(condition)** 성격을 ERC-1155의 position ID로 인코딩한다. 각 position ID는 `(collateralToken, conditionId, indexSet)`의 해시로 uniquely 결정된다.

### 2. NegRiskExchange (Negative Risk용 교환 컨트랙트)

[[개념/Negative-Risk]] 마켓 전용. 3개 이상의 결과가 있는 마켓에서资本 효율적인 포지션 구성을 지원한다.

**일반 Yes/No vs Negative-Risk 비교:**

| 항목 | 일반 (Yes/No) | Negative-Risk |
|------|---------------|---------------|
| 결과 수 | 2 | 3개 이상 |
| Split | collateral → Yes + No | collateral → A + B + C + ... |
| Exchange | 기본 Exchange | NegRiskExchange |
| 포지션 조합 | 단순 | 다중 결과 조합 가능 |

Negative-Risk 마켓에서는 여러 결과 중 하나를 선택하는 베팅도 가능하고, "A 또는 B"와 같은 조합 포지션도 토큰화된다. CTF의 partition 메커니즘이 이를 가능하게 한다.

### 3. Proxy/Safe 컨트랙트

각 사용자는 개인 Proxy/Safe 컨트랙트를 통해 거래한다.

**Proxy 컨트랙트의 역할:**

- **Gasless 트랜잭션**: 사용자가 직접 gas를 지불하지 않고, Polymarket이 gas를 메타트랜잭션으로 대신 지불
- **일괄 승인**: 토큰 spend를 컨트랙트마다 개별 승인하지 않고 Proxy 수준에서 관리
- **보안**: Safe 멀티시그 구조로 자금 보호

```
사용자 지갑 → MetaTx 서명 → Polymarket Relayer → Proxy 컨트랙트 → CTF/Exchange
```

## Subgraph (The Graph)

Polymarket은 The Graph를 통해 온체인 데이터를 GraphQL로 조회할 수 있는 Subgraph를 제공한다.

**주요 쿼리 타입:**

- `positions`: 사용자의 현재 포지션 조회
- `transfers`: 토큰 이동 이력 (매수/매도 추적)
- `splits`, `merges`: Split/Merge 이벤트 로깅
- `redemptions`: 정산 및 Redeem 이력

**Subgraph 활용 예시 — 특정 트레이더 포지션 추적:**

```graphql
query TraderPositions($trader: Bytes!) {
  positions(where: { account: $trader }) {
    id
    balance
    positionIds {
      payout
      conditionIds
    }
  }
}
```

Copytrade에서 Subgraph는 **Gamma API보다 더 정확한** 데이터를 제공한다. Gamma API는 오더북 기반 주문 데이터만 보여주지만, Subgraph는 실제 온체인 토큰 잔고와 이동 이력을 그대로 읽을 수 있다.

## 블록체인 데이터 리소스

| 도구 | 용도 | 특징 |
|------|------|------|
| **Dune Analytics** | 쿼리형 대시보드 | SQL로 직접 온체인 데이터 분석 |
| **Polygonscan** | 트랜잭션 탐색기 | 개별 tx 상세, 컨트랙트 소스 코드 |
| **The Graph Subgraph** | GraphQL 쿼리 | 구조화된 데이터 접근 |
| **ethers.js / viem** | 프로그래매틱 접근 | RPC 직접 호출, 커스텀 읽기/쓰기 |

## Copytrade 관점에서의 온체인 데이터 활용

### 1. 실시간 고래 감지 (Whale Watching)

온체인 이벤트를 구독하면 대규모 Split/Merge/Redeem을 **CLOB 체결 전에** 감지할 수 있다.

```python
# WebSocket/Alchemy: 대규모 토큰 이동 감지
from web3 import Web3

CTF_ABI = [{"anonymous": False, "inputs": [...], "name": "TransferSingle"}]

def on_transfer(event):
    value = event['args']['_value']
    if value > LARGE_THRESHOLD:
        log_whale_alert(event)
```

- Gamma API의 order book 데이터와 Subgraph의 온체인 데이터를 결합하면 **선행 신호**로 활용 가능
- 특히 Split 연산이 대규모로 발생하면 해당 트레이더가市场预期 변화를 포지션으로 옮기고 있다는 신호

### 2. 정산 상태 모니터링

이벤트 결과 발표 후 토큰 Redeem 진행률을 온체인에서 추적할 수 있다.

```python
# 특정 마켓의 Redeem 진행률 (유사 코드)
redeemed = query_contract.call('redeemed_shares', condition_id)
total_shares = query_contract.call('split_position', condition_id)
redemption_rate = redeemed / total_shares
```

Copytrade에서 정산 지연은 **기회비용**으로 직결된다. 정산이 늦은 마켓의 자금이 묶이면 다음 기회를 놓치게 된다.

### 3. 컨트랙트 직접 호출로 가스 최적화

Polymarket 프론트엔드를 통하지 않고, 컨트랙트를 직접 호출하면:

- **중간 레이어 제거**: 프론트엔드 API 호출 → 릴레이 → 컨트랙트 vs 직접 컨트랙트 호출
- **배치 처리**: 여러 Split/Merge를 단일 트랜잭션으로 묶어 가스 절감
- **커스텀 로직**: Copytrade 알고리즘에서 조건부 실행 (예: 특정 슬리피지 임계치 초과 시 자동 취소)

### 4. 데이터 수집 파이프라인

```
Polygon RPC (Infura/Alchemy)
  ├── 이벤트 리스너 (Transfer, Split, Merge, Redeem)
  │       └── 실시간 DB 저장 → Discord/Webhook 알림
  │
  ├── Subgraph GraphQL (주기적 폴링)
  │       └── 트레이더 포지션 스냅샷 → 분석 DB
  │
  └── Dune API (배치 쿼리)
          └── 일간/주간 요약 리포트
```

Copytrade 시스템은 이 세 가지 계층을 조합한다: 실시간 감지는 이벤트 리스너, 포지션 분석은 Subgraph, 거시적 트렌드는 Dune.

## 보안 및 검증 포인트

### 오라클 리스크

Polymarket의 결과 정산은 외부 오라클에 의존한다. 오라클이 잘못된 결과를 보고하면 해당 마켓의 토큰 가치가 왜곡된다.

- UMA(University of Massachusetts) 오라클을 사용하는 마켓과 자체 정산 마켓 구분 필요
- [[개념/Negative-Risk]] 마켓은 다중 결과 간 오라클 일관성이 특히 중요

### 컨트랙트 업그레이드 가능성

- Proxy 패턴을 사용하는 컨트랙트는 업그레이드 가능
- Polymarket이 컨트랙트 로직을 변경할 수 있음을 인지하고, 주요 변경사항은 [[개념/Polymarket]]의 공식 채널에서 확인해야 함

### 접근 제어 (Access Control)

- 일부 컨트랙트 함수는 `onlyOwner` 또는 특정 역할(role)로 제한됨
- 정산 컨트랙트에 대한 접근 권한 변화는 마켓 결과에 직접적 영향
